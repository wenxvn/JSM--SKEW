# SKEW 项目协作规范

## 语言与文档

- 本文件、`prompts/` 下的实施提示词、设计说明、运行日志、变更记录、测试记录及其他项目记录文件必须使用中文。
- 代码标识符、第三方服务名、命令、环境变量及不可翻译的技术术语可保留英文；说明文字必须为中文。
- 不要在本文档或代码中硬编码新闻源 URL、密钥、令牌或其他敏感信息。

## 产品范围

SKEW 从配置好的新闻源收集真实新闻，使用 AI 完成文章分析，将结果持久化到 Supabase，并以适合阅读的方式展示情绪与框架（framing）洞察。

当前只构建以下能力，不要无关扩展：

- 带新闻卡片的首页。
- 展示完整文章分析的新闻详情页。
- Clerk 身份认证与受保护路由。
- Supabase 数据持久化。
- Oxylabs 抓取与 Oxylabs Scheduler 调度。
- AI 文章分析。
- 运行日志。
- 使用 pgvector 查询相似文章。
- 使用 Vercel Cron 触发自动调度。
- 简洁、响应式的界面。
- 使用 PostHog 进行隐私合规的产品事件、功能使用和运行观测。

## 每次实施请求的工作流

除非用户明确表示可以跳过提示词创建，否则每次实施请求必须按以下五个阶段执行：

1. 规划：阅读本 `AGENTS.md`，确认任务范围、目标环境和风险等级；选择并完整阅读与任务相关的 `skills/<名称>/SKILL.md`。用户明确指定的技能必须优先阅读，无关技能不得套用。
2. 设计：检查相关代码、依赖、迁移、配置和已有记录；仅在需求存在实质性歧义且无法从上下文确认时，提出一个聚焦的问题；在 `prompts/` 创建详细的中文实施提示词。
3. 批准：明确请求用户批准：`我已在 prompts/<文件名>.md 准备好实施提示词，可以开始执行吗？`。只有收到批准后才修改实现、数据库结构、受版本控制配置或外部服务配置。
4. 实施：按已批准提示词执行最小且聚焦的改动。涉及生产环境、数据、权限、成本或不可逆操作时，必须在实际执行前再次取得针对该操作的明确批准。
5. 验证与同步：运行相关检查，验证日志、关键功能、回滚路径和环境边界；用中文记录实际外部操作与结果，并更新长期规范、范围或变更记录。

## 本地技能使用

本项目的本地工作流技能以仓库 `skills/` 为准。使用技能前必须读取其 `SKILL.md`，并遵守该技能的产物归属与边界：

- `skills/scope`：将产品想法或下一阶段工作整理为长期范围和功能顺序。
- `skills/architect`：处理技术选型、跨层约束和需要明确决策的设计规格。
- `skills/develop`：按照已批准的规格或提示词实施功能。
- `skills/check`：审查改动或验证实现与规格的一致性。
- `skills/test`：建立或执行与变更相称的测试策略。
- `skills/debug`：定位、复现并修复已报告的故障。
- `skills/audit`：审计既有架构、约定、安全性或根级长期规范。
- `skills/document`：基于提交、差异或已知事实编写发布、变更和事故记录。
- `skills/sync`：在改动完成后同步维护长期上下文、范围状态和规格状态。

Next.js、路由、服务端/客户端边界、API Route 和界面模式优先查阅 `node_modules/next/dist/docs/`。Cheerio、Zod、Tailwind 和 shadcn/ui 使用项目既有模式、相应包文档以及 Next.js 本地文档；不要自行发明技能或假定不存在的 `.agents/skills/` 路径可用。

## 已安装的集成技能

以下 `.agents/skills/` 技能已随项目提交，用于补充第三方服务的官方实现约定。它们不是 `skills/` 下的本地工作流技能，也不改变本文件规定的环境隔离、审批和密钥边界。仅在任务涉及对应服务时完整阅读其 `SKILL.md`：

- `.agents/skills/ai-sdk`：Vercel AI SDK、Provider、结构化输出和嵌入。
- `.agents/skills/clerk`、`.agents/skills/clerk-setup`、`.agents/skills/clerk-nextjs-patterns`：Clerk 基础接入、Next.js 路由保护和服务端认证模式。
- `.agents/skills/clerk-backend-api`：Clerk 后端管理 API；调用任何用户、组织或应用管理操作前仍须取得本文件规定的批准。
- `.agents/skills/clerk-testing`：Clerk 集成测试。
- `.agents/skills/supabase`、`.agents/skills/supabase-postgres-best-practices`：Supabase 客户端、迁移、RLS 与 Postgres 安全实践。
- `.agents/skills/oxylabs-web-scraper`：Oxylabs Web Scraper API；使用前仍须确认来源范围、数量、成本和目标环境。

新增、更新或删除项目级技能时，必须同步更新 `skills-lock.json` 和本节；不得把任何凭证写入技能文件或锁文件。

## 实施提示词文件

- 所有实施提示词放在 `prompts/`，使用语义明确的 kebab-case 文件名，例如 `prompts/oxylabs-scraping.md`、`prompts/ai-analysis.md`、`prompts/news-details-page-ui.md`。
- 每个提示词必须包含：目标、已阅读的技能、已检查的代码、决策与假设、预计变更的文件、实施要求、安全要求、验收标准、需运行的检查，以及实施后的精确手动测试步骤。
- 界面任务还必须说明视觉解读、布局、字体层级、间距、颜色、响应式行为与像素级验收期望。

## 架构边界

以下层必须保持分离：

- 网站层：页面、卡片、详情界面和认证界面。
- API 层：只保留轻量 Route Handler，负责请求校验、授权和调用应用服务。
- 数据库层：所有 Supabase 读写。
- 抓取层：Oxylabs 调用与 Scheduler 集成。
- 解析层：文章链接提取、清理和文章有效性校验。
- AI 层：文章分析与输出校验。
- 管道层：抓取/分析编排和日志追踪。
- 向量层：文章嵌入存储和 pgvector 相似度查询。
- 可观测层：PostHog 负责隐私合规的产品事件、功能使用、错误和性能观测。

界面只能展示已存储的数据，绝不能直接抓取、执行 AI 分析或修改管道状态。

## 外部服务与控制台维护

外部服务网站控制台仅用于配置、只读检查和运维。控制台不能替代受版本控制的代码、迁移或配置说明；所有实际控制台变更均须在中文记录中说明时间、环境、操作者、影响、结果、关联链接或运行 ID 以及后续事项，且不得记录凭证。

- Supabase：业务数据的唯一事实来源，负责数据库、迁移、RLS、查询、日志和 pgvector。可进行只读状态检查；生产写入、删除、批量修复、RLS/权限变更或 pgvector 重建必须先获明确批准。数据库变更必须具有可审阅迁移，并提供回滚或恢复说明。
- Clerk：身份、会话和认证状态的唯一事实来源。可进行只读配置检查；生产用户、组织、角色、Webhook、重定向 URL、权限或密钥变更必须先获明确批准。业务权限必须在服务端复核，不得只依赖客户端界面。
- Oxylabs：外部新闻抓取及 Scheduler 任务的执行服务。可检查历史运行；创建、启停、修改或删除调度任务，以及对真实来源发起抓取，必须先确认来源范围、单来源文章数量、成本、目标环境和预期输出。
- PostHog：仅用于匿名化产品分析和运行观测，不得作为业务数据、认证状态或授权判断的来源。可查看仪表盘和事件健康度；生产项目、权限、数据删除、保留期、数据导出或事件收集策略变更必须先获明确批准。

环境未明确时，先询问并默认不触碰生产环境。每次外部服务操作前确认目标环境；开发、预发布和生产环境必须使用彼此隔离的项目、数据和凭证。

## Supabase 数据约束

- Supabase 是应用数据的唯一事实来源。
- 核心表为：`sources`、`articles`、`article_analyses`、`logs`、`oxylabs_schedules`、`oxylabs_schedule_runs`。
- 抓取逻辑必须从 `sources` 读取启用的新闻源；不得在抓取逻辑或本文件中硬编码来源 URL。
- 每个新闻源至少保存抓取所需的：名称、首页/列表页 URL（`listing_url`）、可选解析策略、启用状态，以及可选的其他抓取配置。
- 通过控制台进行的 schema、索引、RLS 或函数变更，必须同步为受版本控制的迁移或配置说明，不得只保留在控制台中。

## 抓取与入库流程

在实现或运行抓取前，先读取 Supabase 中启用的新闻源并向用户展示可用来源名称；询问要抓取哪些来源以及每个来源的文章数量。

- 如果用户已明确数量或范围，按其要求匹配启用来源。
- 如果用户未指定，默认选择全部启用来源和系统定义的单来源默认上限。
- 不得虚构来源 URL，不得抓取 Supabase 中未保存的子端点。
- `sources.listing_url` 仅是新闻源首页或列表入口，不能作为文章保存。

手动抓取和 Scheduler 抓取必须使用同一条规范化管道；两者只可在触发方式和首页 HTML 的来源上不同：手动抓取通过 Oxylabs 获取已存储首页，调度抓取使用已完成 Oxylabs 任务的结果。

管道必须依序完成：

1. 读取选中的启用新闻源。
2. 获取每个来源的首页 HTML，且不沿子链接继续寻找列表页。
3. 仅从可见的首页新闻卡片提取候选链接。
4. 在详情抓取前排除非文章链接。
5. 规范化与去重候选 URL，并通过 URL 存在性检查跳过已入库 URL。
6. 只抓取通过候选 URL 校验的文章详情页。
7. 校验并清理详情页内容，必须通过文章内容门槛。
8. 仅追加插入有效文章；绝不将来源首页、列表页或分类页保存为文章。
9. 在运行过程中写入日志，并输出最终汇总对象。

## 安全与质量

- 认证、授权、输入校验、服务端密钥隔离和错误处理必须在服务端边界完成。
- 所有外部服务响应、抓取内容与 AI 输出均视为不可信输入，入库或展示前必须校验。
- 数据库写入要具备 URL 去重和可追踪日志；失败不得破坏既有数据。
- 密钥、服务角色凭证、Cookie、访问令牌和 Webhook 密钥只能存在于受控环境变量和服务端；不得提交、输出、截图记录或复制到提示词、日志和聊天回复中。
- 外部服务凭证与角色遵循最小权限原则，并按开发、预发布和生产环境隔离；密钥轮换、生产权限调整和环境切换必须先获明确批准。
- PostHog 事件必须先定义名称、用途、字段与保留边界。默认不得发送文章全文、密钥、Cookie、访问令牌、邮箱或其他直接标识信息；如确有必要，必须先获得用户对合规方案的明确批准。
- 在发布或外部服务变更前后，检查目标环境、错误日志、关键事件、成本影响和回滚路径；任何失败都必须记录原因、影响和恢复状态。
- 保持改动小而聚焦，不因未来猜测增加功能。
