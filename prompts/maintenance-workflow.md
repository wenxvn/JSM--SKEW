# 完善项目维护工作流

## 目标

更新根目录 `AGENTS.md`，使后续代理能够使用仓库 `skills/` 中真实存在的技能，并安全地通过 Supabase、Clerk、Oxylabs 和 PostHog 的网站控制台维护 SKEW 项目。新增规则必须保持中文、可执行且避免不必要的流程膨胀。

## 已阅读的技能

- `skills/sync/SKILL.md`：确认 `AGENTS.md` 是工具无关的长期上下文；维护应使用小范围、可审阅的编辑。
- `skills/architect/SKILL.md`：确认架构决策、实施规格与实现应由不同流程负责，并且本地技能目录是项目能力的实际来源。
- `skills/document/SKILL.md`：确认记录必须基于实际证据，不能编造已发生的操作或结果。

## 已检查的代码与配置

- 根目录 `AGENTS.md`：已定义中文记录、提示词批准、产品范围、分层架构、Supabase 规则和抓取流程。
- `skills/`：实际包含 `architect`、`audit`、`check`、`debug`、`develop`、`document`、`scope`、`sync`、`test` 九个技能。
- `.agents/skills/`：当前不存在；因此现有对 `.agents/skills/clerk`、`.agents/skills/supabase`、`.agents/skills/oxylabs-web-scraper`、`.agents/skills/ai-sdk` 的路径引用不可执行。
- 项目当前为最小 Next.js 工程，尚未发现 Supabase、Clerk、Oxylabs 或 PostHog 的集成配置。

## 决策与假设

- 以仓库 `skills/` 为唯一的本地工作流技能目录；不得再把不存在的 `.agents/skills/` 当作可用依赖。
- 外部服务的网站控制台可以用于配置、检查和运维，但所有生产性、不可逆或影响数据/用户的操作仍须在执行前取得用户明确批准。
- Supabase 是业务数据的唯一事实来源；Clerk 是身份和会话的唯一事实来源；Oxylabs 是外部抓取任务的执行服务；PostHog 仅用于匿名化产品分析和运行观测，不存放业务事实或凭证。
- 当前没有可用的专用外部服务技能。实现外部服务集成时，优先使用对应官方文档和项目已安装依赖；如果未来添加技能，先记录其路径和适用范围，再使用。
- 不引入新的服务、SDK、数据表、事件或自动化任务，除非后续具体实施提示词已获批准。

## 预计变更的文件

- `AGENTS.md`：仅更新项目工作流和长期维护约束。
- `prompts/maintenance-workflow.md`：本提示词，保留为本次规范变更的中文记录。

## 实施要求

1. 将“技能使用边界”改为反映真实 `skills/` 目录，并定义各技能的使用时机：范围规划、架构决策、开发、检查、测试、调试、审计、文档和同步。
2. 在每次任务工作流中明确：先选择相关本地技能，读取其 `SKILL.md`，再按技能职责产出和执行；无关技能不得读取或套用。
3. 在产品范围和架构边界中补充 PostHog，限定其职责为产品事件、功能使用、错误/性能观测和隐私合规的分析，不得充当业务数据源或认证系统。
4. 新增“外部服务与控制台维护”章节，分别规定 Supabase、Clerk、Oxylabs、PostHog 的职责、允许的日常检查、必须先获批准的操作，以及控制台变更后的中文记录要求。
5. 新增配置与密钥规则：只在服务端和受控环境变量中使用密钥；不输出、提交、截图记录或复制密钥；区分开发、预发布和生产环境；最小权限配置。
6. 新增数据库安全规则：迁移必须可审阅、可回滚或具备恢复说明；生产写入、删除、RLS/权限变更、批量修复和 pgvector 重建前必须获得明确批准；外部控制台改动必须同步到迁移或受版本控制的配置说明。
7. 新增认证安全规则：Clerk 的用户、组织、角色、Webhook、重定向 URL 或生产密钥变更均须先确认；业务权限必须在服务端复核，不以客户端显示为准。
8. 新增抓取运行规则：创建、启停、修改或删除 Oxylabs 调度任务，及对真实来源发起抓取，均须先确认范围、单来源数量、成本和目标环境；记录运行 ID、来源、数量、结果与失败原因，但不记录凭证。
9. 新增 PostHog 隐私和事件规则：事件先定义名称、用途、字段和保留边界；禁止发送文章全文、密钥、Cookie、访问令牌、邮箱等直接标识信息，除非用户已明确批准合规方案；生产项目、数据删除、保留期、权限和导出设置变更须先确认。
10. 新增验证与交接规则：变更前后检查环境、日志、错误率、关键事件和回滚路径；每项实际外部操作在中文记录中写明时间、环境、操作者、影响、结果、链接/ID 和后续事项。
11. 保留当前“先创建提示词，用户批准后再实现”的规则，并将常规工作流细化为“规划、设计、实施、验证、同步”五个阶段，避免重复或相互矛盾。

## 安全要求

- 不执行真实生产操作、数据删除、批量数据写入、权限变更、密钥轮换、收费抓取、调度启停或遥测数据导出。
- 不读取或展示环境变量中的敏感值，不将其写入仓库、提示词、日志或聊天回复。
- 所有来自抓取服务、Webhook、浏览器和 AI 的输入在服务端校验。
- 仅在用户指定的环境操作；环境未明确时，先询问，默认不触碰生产环境。
- 外部服务控制台的手工变更必须有可追踪的中文记录和可复现的受版本控制配置/迁移说明。

## 验收标准

- `AGENTS.md` 全文使用中文，且明确其他记录文件使用中文。
- 技能目录引用与仓库实际结构一致，九个本地技能均有清晰职责。
- Supabase、Clerk、Oxylabs、PostHog 的职责、数据边界、隐私规则和批准门槛明确且互不冲突。
- 日常只读检查与有副作用的控制台操作有明确区分。
- 所有高风险或生产性操作都要求用户先批准，且未添加任何真实凭证或具体来源 URL。
- 原有新闻抓取、入库和界面只读约束保持有效。

## 需运行的检查

1. 检查 `AGENTS.md` 的标题结构与中文要求。
2. 搜索不存在的 `.agents/skills/` 引用，确认已移除或明确标注为不可用。
3. 检查 `AGENTS.md` 中 Supabase、Clerk、Oxylabs、PostHog 与九个本地技能均被提及。
4. 执行 `git diff --check`，确认无空白错误。

## 实施后的手动测试步骤

1. 以一个“新增认证功能”的请求验证流程会选择 `skills/architect`、`skills/develop`、`skills/check` 和 `skills/test` 的相关规则。
2. 以一个“检查 Supabase 表状态”的请求验证流程允许只读检查，但不会执行写入。
3. 以一个“删除生产 PostHog 项目”或“启动真实 Oxylabs 抓取”的请求验证流程会先要求用户确认。
4. 以一个“修改新闻卡片”的请求验证流程仍要求先在 `prompts/` 创建中文实施提示词并取得批准。
