# 项目视觉规范长期化实施提示词

## 目标

将已实施的中文新闻视觉风格从一次性实施提示词和 `app/globals.css` 提炼为项目级、可持续维护的设计规范。新增根目录 `design.md` 作为界面设计的唯一长期来源，并在根 `AGENTS.md` 中加入简短且强制的引用规则，确保后续页面、组件和界面改动遵守同一套视觉语言。

本次不是重做首页，也不扩展产品功能。目标是让未来实现者知道应当读取什么、复用什么、不得破坏什么。

## 已阅读的技能

- `skills/audit/SKILL.md`：根 `AGENTS.md` 为项目长期上下文，已有文件只能做保守、明确的补充，不能重写既有人工维护的内容。
- `skills/audit/modes/gapfill.md`：确认本次是既有根规则文件的缺口补充，新增规则应先明确提出且仅加入所需内容。
- `skills/sync/SKILL.md`：确认新建项目级 `design.md` 后，允许在最近的 `AGENTS.md` 中加入一行设计系统指针。

## 已检查的文件

- `AGENTS.md`：现有项目级规范已定义语言、产品范围、工作流、架构边界和安全要求，但没有可被后续界面任务引用的设计系统入口。
- `app/globals.css`：已包含 `--skew-*` 颜色、间距、圆角、阴影和字体令牌，以及响应式阅读界面样式。
- `app/layout.tsx`：根文档语言已是 `zh-CN`，页面使用系统中英文字体栈，不依赖远程 Google Fonts。
- `app/page.tsx`：首页已采用黑白编辑式排版、细边框、红蓝视角语义和真实无数据状态。
- `prompts/chinese-news-visual-system.md`：记录本次视觉实现的完整决策，但它是实施历史而不是未来页面的长期规范入口。
- 仓库中不存在项目级 `design.md` 或其他自定义界面规则文件。

## 决策与假设

- 在仓库根目录新增 `design.md`，而不是将大量视觉细节塞进 `AGENTS.md`。`AGENTS.md` 保持简短，承担发现和强制阅读的职责；`design.md` 保存具体设计语言。
- `design.md` 使用中文，且只记录 SKEW 当前已确认的视觉约束。它不包含密钥、新闻源 URL、虚构内容、第三方服务配置或一次性实施过程。
- `design.md` 的内容以当前 `app/globals.css` 为准。若未来设计令牌改变，必须同时更新 CSS 与该文档；不得让文档与实现各自演进。
- 在 `AGENTS.md` 的语言与文档规则附近新增一个“界面设计规范”小节，明确要求任何页面、组件、全局样式和界面相关实施提示词在设计或实现前阅读 `design.md`，并遵守其中的令牌、排版和响应式约束。
- 不修改 `skills/`、`.agents/skills/`、`node_modules/` 或外部服务配置。它们不是本项目设计规范的归属位置。

## 预计变更的文件

- `design.md`：新增项目级中文新闻界面设计规范。
- `AGENTS.md`：新增简短的项目级设计规范入口与遵循要求。

不修改应用代码、依赖、数据库、环境变量、迁移、认证配置、外部服务配置或生产环境。

## 实施要求

### `design.md` 的结构与内容

文档至少包含以下章节，并将每项写成可执行的约束而不是泛泛描述：

1. **定位与适用范围**：SKEW 是简体中文新闻分析产品，视觉方向是克制、清晰的编辑式阅读界面。设计风格参考已批准的样图语言，不复制其内容或布局。
2. **设计原则**：信息优先、真实数据优先、克制装饰、立场语义不等于事实判断、可访问性与响应式优先。
3. **颜色与语义**：白色主画布、近黑正文、浅灰表面和边界；红色只表示偏左，蓝色只表示偏右，中性灰表示中立；禁止将红蓝用作无语义装饰；列出当前 CSS 变量和值的来源。
4. **排版**：根语言为 `zh-CN`；优先系统中文无衬线字体栈；英文、数字和品牌使用自然回退；定义品牌、页面标题、区块标题、卡片标题、正文、元信息的推荐字号与行高；禁止负字距。
5. **间距、边界与层级**：四像素倍数间距、容器最大宽度约 1280px、细灰边框、4px 或 8px 圆角、轻量阴影。不得使用大圆角、渐变、玻璃拟态、装饰性光球或营销式巨型卡片。
6. **布局与响应式**：宽屏网格、小于约 900px 减列、小于约 640px 单列；最小支持 320px；不可横向滚动、文字重叠或裁切；不得让字号随视口宽度缩放。
7. **组件规则**：卡片、分类标签、偏向条、空状态、按钮与链接的使用原则。偏向条必须同时提供文字标签，静态或未实现的控件不得误导用户为可用功能。
8. **内容与数据边界**：界面只展示已保存的数据；数据层未接通时使用诚实空状态，不用虚构文章、来源、统计数或人物图片填充页面。
9. **实现与验证清单**：优先复用 `app/globals.css` 的 `--skew-*` 令牌；不在 JSX 或局部样式散落重复色值；界面改动运行 lint 与 build，并在宽屏、390px 和 320px 视口进行视觉检查。
10. **变更规则**：任何影响令牌、组件基调或响应式策略的改动必须在同一变更中更新 `design.md`、`app/globals.css` 与相关验收记录。

### `AGENTS.md` 的补充方式

- 只增加一个简短的“界面设计规范”小节，不改写或移动既有段落。
- 小节必须链接到根目录 `design.md`，并说明界面任务应在设计和实现前完整阅读该文件。
- 明确 `design.md` 是界面视觉规范的长期来源，`app/globals.css` 是令牌的运行时实现；两者在设计变更中必须同步。
- 保留现有的中文文档要求、产品范围、认证和数据边界，新增内容不得削弱这些限制。

## 安全要求

- 不写入密钥、令牌、Cookie、个人信息、新闻源 URL、Supabase 数据或第三方控制台信息。
- 不执行外部服务调用、数据库操作、抓取、AI 分析、PostHog 事件、部署或认证配置变更。
- 不复制参考图中的新闻内容、人物图像、品牌素材或完整页面结构。

## 验收标准

1. 根目录存在中文 `design.md`，且包含颜色、字体、间距、布局、组件、数据边界、响应式、无障碍和变更同步规则。
2. `design.md` 明确引用 `app/globals.css` 的 `--skew-*` 令牌，不创建与现有 CSS 冲突的第二套设计值。
3. `AGENTS.md` 有清晰且简短的设计规范入口，未来界面任务无需翻找历史提示词即可知道必须阅读 `design.md`。
4. 既有产品安全边界、数据真实性规则和中文文档要求未被删除、改写或弱化。
5. 不产生任何应用代码、依赖、数据库、环境变量或外部服务变更。
6. Markdown 链接可解析，`git diff --check` 无空白错误。

## 需运行的检查

1. `git diff --check`
2. 检查 `AGENTS.md` 到 `design.md` 的 Markdown 相对链接。
3. 用搜索确认 `design.md` 的关键令牌名、中文排版约束、红蓝语义和响应式阈值均存在。
4. 确认差异只包含 `design.md`、`AGENTS.md` 和本实施提示词，没有应用代码或外部配置文件。

## 实施后的精确手动测试步骤

1. 从仓库根目录打开 `AGENTS.md`，确认可以直接定位“界面设计规范”并跳转到 `design.md`。
2. 通读 `design.md`，确认新页面设计者可以据此确定颜色语义、字体、字阶、间距、圆角、响应式断点与禁止项。
3. 对照 `app/globals.css`，确认 `design.md` 中的令牌和色值描述与 `--skew-*` 实现一致。
4. 检查 Git 状态，确认未出现外部服务配置、密钥、数据库迁移或应用功能代码的意外变更。

## 回滚路径

本次仅新增 `design.md` 并向 `AGENTS.md` 添加一个小节。若规范表述需要调整，恢复或删除这两个文档变更即可；不涉及运行数据、部署或外部服务回滚。
