# 中文新闻视觉系统实施提示词

## 目标

为 SKEW 建立适合中文新闻阅读产品的基础视觉系统，并将当前受保护首页从开发占位卡片调整为可持续扩展的中文新闻首页外壳。视觉方向参考用户提供的样图，但不复制样图中的页面内容、模块或英文文本。

目标是吸收其克制的编辑感：高对比黑白中性色、细边框、大留白、清晰的网格、紧凑的信息层级，以及仅用于立场语义的红蓝强调色。界面应服务于新闻阅读和分析，而不是成为设计系统展示页。

- `skills/test/SKILL.md`：确认样式文件本身不应引入为覆盖率而写的测试，实施后以 lint、生产构建和浏览器视觉检查验证。

- `app/page.tsx`：当前首页保持 Clerk 服务端认证保护，但只显示开发环境占位内容。

## 决策与假设

- 页面主语言为简体中文，品牌名称保留 `SKEW`。将根文档语言设为 `zh-CN`。
- 不增加远程字体请求或第三方包。英文、数字和中文均使用系统可用的无衬线字体栈，优先覆盖 `PingFang SC`、`Hiragino Sans GB`、`Microsoft YaHei`、`Noto Sans CJK SC` 等中文字形，并以通用 Latin 字体回退，以保持离线构建稳定和中文可读性。实施中移除项目原有的 `next/font/google` Geist 加载器，因为即使在允许网络访问后构建也无法取得字体资源。
- 使用白色主画布、接近黑色正文、浅灰表面和极浅灰边界。红色仅表达偏左，蓝色仅表达偏右，中性灰表达居中；不得用红蓝作为无语义的装饰色。
- 关闭当前按系统自动切换深色主题的规则，保证新闻分析的语义色在浅色阅读界面上始终一致。日后若需要深色模式，必须单独设计对比度和立场色映射。
- 采用四像素倍数的间距令牌和八像素以内的卡片圆角。布局容器最大宽度为约 1280px，桌面端以双列或多列网格组织信息，小屏端收敛为单列。
- 当前仓库没有可读取的已存储文章数据。首页不得使用虚构新闻卡片假装真实数据；实现中文品牌头部、页面标题、筛选外壳与明确的空状态，待文章查询层完成后再由真实存储数据驱动新闻卡片。
- 保持现有的服务端 `auth()` 路由保护。实现不能把 Clerk 状态、密钥或任何用户数据传入样式层，也不改变登录流程。

## 预计变更的文件

- `app/globals.css`：替换默认主题，定义颜色、间距、字阶、边框、阴影、容器、标签、按钮、卡片、偏向仪表和空状态的全局 CSS 令牌与基础样式。
- `app/layout.tsx`：将文档语言调整为 `zh-CN`，保留现有 Next.js 字体变量和 ClerkProvider。
- `app/page.tsx`：保留认证流程，替换开发占位内容为中文首页结构，使用全局样式类展示品牌、导航骨架、新闻浏览标题、筛选区域与无文章空状态。

除上述文件外，不修改数据库迁移、环境变量、外部服务配置、认证配置或依赖清单。

## 实施要求

### 视觉令牌

- 在 `:root` 中定义语义化 CSS 变量，而不是把颜色散落在 JSX 中。至少覆盖主背景、次级背景、主文本、次级文本、边框、分隔线、偏左、居中、偏右、焦点环和阴影。
- 颜色建议以样图为基准但允许为中文阅读微调：正文约 `#0D0D0F`，次级文字约 `#6E7280`，主背景 `#FFFFFF`，次级背景约 `#F7F7F7`，边框约 `#E5E7EB`，偏左约 `#B42318`，偏右约 `#1D4ED8`，中性区域使用低对比灰。
- 为 Tailwind CSS 4 的 `@theme inline` 补充与语义变量对应的颜色和字体别名，使后续组件能在不复制色值的前提下复用它们。
- 全局 `box-sizing`、基础元素间距、选中文本、键盘焦点和按钮禁用状态必须有稳定且可访问的默认表现。

### 中文排版与布局

- 正文使用适合中文的无衬线字体栈，英文、数字和品牌可自然回退至 Geist。不得依赖一款可能缺少中文字形的英文展示字体。
- 字阶应清晰而克制：品牌或页面主标题约 30 至 32px，区块标题约 20 至 24px，卡片标题约 18 至 20px，正文约 15 至 16px，元数据和标签约 12 至 13px。中文行高应为 1.5 至 1.7，标题行高约 1.25 至 1.35。
- 禁止使用负字距。中文标签和导航以正常字距呈现，避免为了模仿英文样图而挤压汉字。
- 页面使用居中容器与稳定的水平内边距。桌面端保留明显的页面留白，移动端将内边距缩小但不可使文字紧贴屏幕边缘。
- 使用细实线、轻微阴影和不超过 8px 的圆角表达层级。不得添加渐变、玻璃拟态、大面积彩色背景、装饰性光球或营销式大卡片。

### 首页结构

- 已登录用户进入首页后，保留一个紧凑的品牌头部。左侧突出 `SKEW`，右侧只保留与当前页面相关的文字导航骨架或状态，不伪造未实现的可用功能。
- 主内容包含中文页面标题和一段简短说明，使用新闻编辑界面的信息密度，而不是首屏营销文案。
- 预留分类筛选与分析偏向图例的样式位置。未接通真实交互前，控件应以不可误导的静态状态或禁用状态呈现，不能暗示筛选已生效。
- 无文章时显示语义清晰的中文空状态，说明当前还没有可展示的已存储分析结果。不要提供不存在的新闻内容、人物照片、来源名称或统计数据。
- 为未来真实文章卡片定义可复用类：来源和时间元信息、标题、摘要、框架标签、以及三段式偏向条。偏向条必须有文本标签或辅助说明，不能只靠红蓝颜色传达含义。

### 响应式与可访问性

- 宽屏时容器不超过约 1280px，内容网格可使用多列；在约 900px 以下减少列数，在约 640px 以下为单列。
- 顶部导航、标题、筛选区域和空状态在 320px 至宽桌面范围内不得溢出、重叠或产生水平滚动。
- 可交互元素使用原生按钮或链接，具有可见焦点状态、足够点击面积和符合正文对比度的文字颜色。
- 偏左、居中、偏右文字和色块应满足可读对比度；不得仅靠颜色表达立场。
- 不使用直接抓取、客户端 AI 调用或客户端数据库写入。页面只处理已有认证结果和本地展示结构。

## 安全要求

- 不读取、输出、写入或提交任何密钥、Cookie、令牌、环境变量值、用户资料或外部服务响应。
- 不执行 Supabase 写入、Clerk 管理、Oxylabs 抓取、AI 分析、PostHog 事件发送、Vercel 部署或任何生产操作。
- 保持认证判断在服务端 `app/page.tsx`，不把授权责任放到 CSS 或仅客户端界面。
- 不新增虚构文章数据。文章卡片的实际数据必须在后续经批准的 Supabase 查询层完成后才接入。

## 验收标准

1. `app/globals.css` 含有可复用的中文新闻视觉令牌，且默认深色模式规则已移除。
2. 根文档 `lang` 为 `zh-CN`，而品牌 `SKEW`、英文和数字仍具有清晰字形，中文正文不会因只使用 Geist 而显示不佳。
3. 登录后的首页不再出现“开发环境已准备就绪”，而是呈现中文新闻产品的品牌、标题和真实空状态。
4. 视觉上体现参考图的黑白编辑感、细灰边界、克制圆角、四像素节奏和红蓝立场语义，但没有复制参考图的内容或页面布局。
5. 窄屏与宽屏下文字无截断、重叠、水平滚动或失去焦点指示。
6. 未登录用户继续进入既有 Clerk 登录流程。
7. 不产生外部服务调用、数据库改动、配置改动或依赖新增。

## 需运行的检查

1. `npm run lint`
2. `npm run build`
3. 启动 `npm run dev` 后，在本地浏览器以已登录和未登录两种状态检查首页与登录重定向。
4. 使用至少一个窄屏视口和一个宽屏视口进行视觉检查，确认布局、文本、焦点与语义色符合验收标准。

## 实施后的精确手动测试步骤

1. 在开发环境启动应用，访问 `http://localhost:3000`，确认未登录访问仍被转至 Clerk 登录页。
2. 使用开发环境测试账号登录，确认首页标题、说明、筛选外壳和空状态均为简体中文，且不再显示开发占位文案。
3. 在宽度 1440px 的视口检查：内容居中、边框细且不抢眼、主标题与正文层级清楚、红蓝只用于偏向语义。
4. 在宽度 390px 和 320px 的视口检查：顶部、标题、筛选区和空状态均为单列或可换行布局，没有横向滚动、裁切或重叠。
5. 用键盘依次移动焦点，确认可操作元素可见焦点明确，禁用或静态筛选项不会误导为可用功能。
6. 打开浏览器网络面板，确认浏览首页不会发起 Oxylabs、AI 或 PostHog 请求，也不会出现 Supabase 写入请求。
